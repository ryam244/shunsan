## Phase 4: アップデート

### 4.1 アップデートの種類

#### OTAアップデート（Over-The-Air）
| 項目 | 内容 |
|------|------|
| 適用範囲 | JavaScript/TypeScriptの変更のみ |
| 方法 | `eas update` コマンド |
| 審査 | 不要 |
| 反映時間 | 即時〜数分 |

#### ストアアップデート
| 項目 | 内容 |
|------|------|
| 適用範囲 | ネイティブコード変更、新SDK、新権限 |
| 方法 | 新ビルド → App Store審査 |
| 審査 | 必要（1-3日） |

### 4.2 バージョニング戦略
```
バージョン形式: MAJOR.MINOR.PATCH
例: 1.0.0 → 1.0.1 → 1.1.0 → 2.0.0

MAJOR: 大幅な機能追加・UI変更
MINOR: 新機能追加
PATCH: バグ修正、軽微な改善
```

### 4.3 アップデートサイクル

| フェーズ | 期間 | 内容 |
|----------|------|------|
| リリース直後 | 1-2週間 | バグ修正集中（v1.0.1, v1.0.2） |
| 安定期 | 月1回 | 機能追加・改善（v1.1.0, v1.2.0） |
| 成熟期 | 四半期 | 大型アップデート（v2.0.0） |

## アクティブユーザー目標

### 獲得戦略
**オーガニック検索のみ**（広告・営業活動なし）

### ユーザー獲得目標（現実的目標）

| フェーズ | 期間 | DAU目標 | MAU目標 | 累計DL | 月間新規DL |
|----------|------|---------|---------|--------|------------|
| ローンチ期 | 1ヶ月目 | 3-5 | 15-30 | 30-50 | 30-50 |
| 初期成長期 | 3ヶ月目 | 8-12 | 50-80 | 150-250 | 50-80 |
| 成長期 | 6ヶ月目 | 20-30 | 100-150 | 400-600 | 80-120 |
| 安定期 | 12ヶ月目 | 50-80 | 300-500 | 1,000-1,500 | 100-150 |
| 拡大期 | 24ヶ月目 | 100-150 | 600-1,000 | 2,500-4,000 | 120-200 |

### なぜこの数値か（根拠）

| 要因 | 説明 |
|------|------|
| **ニッチ市場** | 「不動産仲介営業向け」は検索ボリュームが少ない |
| **競合状況** | 一般的なローン計算アプリは多数存在し埋もれやすい |
| **ASO効果** | 最適化しても新規アプリは月50-100DLが現実的上限 |
| **口コミ効果** | 業界内での評判蓄積に6ヶ月以上かかる |

### 成功の定義（マイルストーン）

| マイルストーン | 条件 | 達成時期目標 |
|----------------|------|--------------|
| 初期成功 | 累計100DL達成 | 2-3ヶ月目 |
| 軌道に乗る | MAU 100達成 | 6ヶ月目 |
| 安定成長 | 月間100DL継続 | 12ヶ月目 |
| 収益化成功 | MRR ¥10,000達成 | 18ヶ月目 |

### KPI定義

| 指標 | 定義 | 目標値 |
|------|------|--------|
| **DAU** | 1日にアプリを開いたユニークユーザー | 上記参照 |
| **MAU** | 1ヶ月にアプリを開いたユニークユーザー | 上記参照 |
| **リテンション（D1）** | 翌日もアプリを使用した率 | 30%+ |
| **リテンション（D7）** | 7日後もアプリを使用した率 | 15%+ |
| **リテンション（D30）** | 30日後もアプリを使用した率 | 8%+ |
| **平均セッション時間** | 1回の利用時間 | 3分+ |
| **計算回数/セッション** | 1セッションで行う計算回数 | 2回+ |
